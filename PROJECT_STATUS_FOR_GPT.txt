Project name:
Vinh Khanh Guide

High-level purpose:
This repository is a tourism guide product centered around "stalls" or points of interest. It includes:
- a .NET MAUI mobile app for end users
- an ASP.NET Core Web API backend
- EF Core with PostgreSQL for persistence
- test projects for application, API, and mobile logic

Current repo shape:
- Main solution: VinhKhanhGuide.sln
- Backend: src/backend/VinhKhanhGuide.Api, Application, Domain, Infrastructure
- Mobile app: src/mobile/VinhKhanhGuide.Mobile
- Tests: tests/VinhKhanhGuide.Application.Tests, tests/VinhKhanhGuide.Api.Tests, tests/VinhKhanhGuide.Mobile.Tests
- There are also side/experimental folders such as TestApp, explan_app, explan_backend, and qr_codes. They do not appear to be the main production path for the Vinh Khanh Guide solution.

What is already implemented in the backend:
- Clean layered structure:
  - Domain contains entities such as Stall and StallTranslation.
  - Application contains use cases and interfaces.
  - Infrastructure contains EF Core persistence, repository implementations, translation stub, and remote-location import support.
  - Api exposes HTTP endpoints.
- Stall data model already exists with fields for:
  - name
  - Vietnamese description
  - latitude / longitude
  - trigger radius
  - priority
  - category
  - open hours
  - image URL and image list JSON
  - address
  - phone
  - reviews JSON
  - map link
  - Vietnamese narration script
  - audio URL
  - active flag
  - average rating
- Stall translations are supported as a separate table/entity with:
  - stall id
  - language code
  - translated name
  - translated description
  - last generated timestamp
- API endpoints currently exposed in StallsController:
  - GET /api/stalls
  - GET /api/stalls/{id}
  - GET /api/stalls/nearby?lat=&lng=&radius=
  - GET /api/stalls/{id}/translation?lang=
  - POST /api/stalls/sync/locations
- Root API health-style endpoint exists:
  - GET /
  - returns a basic "Vinh Khanh Guide API" ready response
- Demo deep-link landing pages exist:
  - GET /demo/open
  - GET /demo/open/stall/{stallId}
  These pages generate HTML that can open the mobile app through the custom URL scheme.
- Database startup behavior:
  - EF Core migrations run automatically on startup when using a relational database.
  - PostgreSQL connection is configured through ConnectionStrings:DefaultConnection.
- Remote location sync pipeline already exists:
  - startup can optionally import/sync location content from a SQL snapshot file
  - manual sync endpoint is available
  - sync logic is designed to insert, update, skip unchanged records, and deactivate missing records
- Translation generation is not connected to a real external provider yet:
  - Infrastructure currently registers a FakeTranslationService
  - this suggests translation behavior is stubbed/faked for now rather than using a live third-party integration

- Main shell navigation already exists with three tabs:
  - List
  - Map
  - Settings

  - shell navigation

  - TTS support
  - device geolocation
  - location tracking

  - offline audio download
  - proximity detection
  - proximity-triggered narration coordination
- App link / deep link handling is already wired:
  - App overrides OnAppLinkRequestReceived
  - external payloads are processed through ExternalEntryService
  - backend demo pages are intended to open the app directly
- Stall list experience:
  - loads cached stalls first when available
  - refreshes live data from API
  - falls back to cache if refresh fails
  - supports narration preview from the list
  - preview language choices currently include vi, en, zh, de
- Stall detail experience:
  - loads a specific stall
  - supports translated display content
  - supports narration playback
  - supports fallback behavior when translated narration/content is unavailable
  - includes offline audio download flow
  - contains logic for external-open/deep-link scenarios and auto-play preparation
- Map experience:
  - loads and displays stalls
  - refreshes device location
  - supports location tracking
  - calculates nearest stall
  - triggers nearby stall prompts
  - coordinates proximity-based narration
  - respects settings such as GPS enabled/disabled
- Settings experience:
  - persists GPS tracking enabled/disabled
  - persists trigger radius multiplier
  - persists auto narration enabled/disabled
  - allows API base URL override
  - allows preferred TTS language selection
  - validates API base URL format
- Offline support already exists in the app design:
  - SQLite-backed stall cache is implemented
  - cached data is used as a fallback when network refresh fails
  - offline/local audio handling is present
- Current bundled mobile app configuration:
  - appsettings.json contains StallApi:BaseUrl as an empty string
  - this is intentional according to MauiProgram comments, so demo/device builds do not silently point to localhost
  - a saved override or environment-specific config is expected to provide the actual backend URL
